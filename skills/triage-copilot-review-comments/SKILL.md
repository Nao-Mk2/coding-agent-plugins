# Triage code reviews

## 目的

レビューされたコメントに対し、今回のpull requestで対応すべきかどうかを判断したレポートを作成する。
最終判断はユーザーが行うため、コード修正は行わなくて良い。
また、ユーザーが判断不要としたコメントについては自動で返信を行う。

## レビュー対象
なければ現在のブランチに対して作成されている pull request を対象とする。

---

## 全体のフロー

1. Pull request のコメントを取得する
2. 各コメントに対し、後述の指針に従いトリアージする
3. ユーザーにレポートを報告する
4. ユーザーが対応不要としたコメントに対し、pull requestのコメントに対し返信を行う

---

## コメント取得方法

1. `triage-review/scripts/fetch-copilot-reviews.sh` を実行し、レビューを取得する。取得できない場合は処理を中断し、ユーザーにエラーを報告する。

## 指針

### 対応の優先順位

次の順で考え、上ほど優先順位が高い。

1. **バグになりそうな点**
   - 条件分岐ミス、境界条件、戻り値の扱い、例外/失敗の見落とし

2. **失敗時の後片付けが破綻しないか（必ず終わるか）**
   - 途中でreturnしても、解放・取り消し・終了処理が漏れないか
   - 「成功後に失敗処理を走らせる」など、失敗時の処理順で誤解を招かないか
   - 長いループや複数段の処理で、リソース（接続・行イテレータ・ファイル等）をため込まないか

3. **データ不整合・失敗時の復旧**
   - 途中失敗で中途半端に残る、再実行で壊れる、二重実行で壊れる
   - 「一部だけ成功」がありうるなら、呼び出し元が誤って成功扱いしないか

4. **並行処理まわり**
   - 並行処理の失敗が呼び出し元に届くか
   - キャンセル/タイムアウト、競合、リーク

5. **運用で困る点（障害対応のしやすさ）**
   - ログが不足して原因が追えない
   - どこまで完了したか分からない
   - 失敗を成功に見せる返し方になっている

6. **テスト不足**
   - 新規ロジックの正常系/異常系、重要分岐、境界条件

---

### 必ず対応してよいこと（Must）

「何が起きるか」と「どう直すか」を短くセットで書く。
さらに可能なら「失敗すると運用でどう困るか」も1文で添える。

**A. 失敗時の後片付け**
- 失敗経路で後片付けが漏れる：解放・取り消し・終了が呼ばれない
- 途中で止まると危ない：ロック保持、接続枯渇、メモリ/ハンドル増加など
- 失敗処理の順が紛らわしい：成功後に失敗処理を試みる、重複して実行される など

**B. 失敗が“成功”に見える**
- エラーを握りつぶす、失敗しても成功扱いになる、結果があいまい
- 「一部成功」がありうるのに、呼び出し元が判断できない

**C. 境界をまたぐ処理の整合**
- ひとつのまとまりとして扱うべき処理が、別の経路・別の実体に分かれてしまう可能性
  - 例：同じ目的の更新なのに、終了条件や適用範囲がずれて混ざる

**D. 監視・調査のための最低限**
- 失敗時に「何が」「どこまで」できたかが追えない
- エラー文言が弱く、再現や切り分けが難しい

---

### 余裕があれば対応してよいこと（Should）

「今直すと得」と言える根拠があるときだけに対応する。

- ログ/コメントの追加：障害時に詰まりそうな箇所だけ（どの場面で困るかも書く）
- 返す値の改善：呼び出し側が誤解しない形に寄せる（成功/失敗/一部成功の区別など）
- テストの追加提案：追加範囲が小さく、効果が明確なとき

---

### 原則対応しないこと（Don’t）

次は基本的に対応しない対応するなら「具体的な不具合」か「運用上の困りごと」と結びつける。

- 書き方の簡潔化だけ（挙動・安全性・運用が変わらない）
- 好みの整形（命名の趣味、空行、コメントの言い回し）
- 一般論としての抽象化・定数化（今回の差分と関係が薄い）
- “きれいにする”だけが目的の提案

---

### 設計・要件に触れるレビューの扱い方（注意）

設計や要件に踏み込む話は、PR内で結論が出にくいことがある。
その場合は「断定」ではなく次の形で書く：

- まず **確認したい前提**（仕様としてOKか）
- 次に **この差分の範囲でできる安全策**（ログ、返り値、後片付けの徹底、失敗時の扱いの明確化 など）

---

### 費用対効果が高い提案を狙う判断基準

次のどれかに当てはまるなら対応する。

- 放置すると事故（障害・データ破損・再実行不能）になりそう
- 放置すると障害時の原因調査が難しい（何がどこまで起きたか分からない）
- 今回の差分だけで小さく直せる（大きな設計変更を要求しない）
- 修正が入ることで挙動が明確に良くなる（好みではない）

---

### 対応の粒度

- 1コメントは1論点。
- 「全面リファクタ」ではなく、この差分の範囲で直せる案を出す。

---

## 出力形式
- 各コメントに対し以下の推奨レベルを判断する
   - Must 🟥: 今回の差分で必ず対応すべき
   - Should 🟨: 余裕があれば今回の差分で対応すべき
   - Don’t 🟩: 対応不要
- レポートはコメントID順に並べる
- レポート形式
   - [推奨レベル]
   - {{コメントのタイトル（コメントから抜き出すか、相応しいものがなければ短いタイトルにコメントを要約したもの）}}
   - {{コメントへのURLリンク}}
   - 理由: {{推奨レベルの判断理由（1行で完結に）}}

---

## 補足
- Test Coverage Report は無視すること。

## コメントへの返信方法

対応不要と判断したものは、コメントへのリプライとしてその旨の回答を入れること。
シンプルに対応しない理由を述べるだけで良い。
