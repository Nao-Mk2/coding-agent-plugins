# Triage & Fix Copilot Review Comments

## 目的

GitHub Copilot のレビューコメントをトリアージし、対応すべきものをコメント1件ごとに Sub-agent で修正・コミットして、最後に全コメントへ返信・resolve まで完結させる。

## レビュー対象

指定がなければ現在のブランチに紐づく Pull Request を対象とする。

---

## 全体のフロー

以下を順に実行する。フェーズ 2 → 3 の間でユーザー確認を挟む。

### フェーズ 1: コメント取得

`triage-copilot-review-comments/scripts/fetch-copilot-reviews.sh` を実行してコメントを取得する。
取得できなければ処理を中断してエラーを報告する。

### フェーズ 2: トリアージ

取得したコメントを後述の指針に従い分類し、ユーザーへレポートを提示する。

**ユーザー確認**: レポートを提示し、修正対象から除外したいコメントがあれば確認する。
確認後、対象コメントリスト（comment URL・discussion_r ID・ファイルパス・行番号・コメント本文・トリアージ理由）を確定する。

### フェーズ 3: Sub-agent による修正・コミット（逐次実行）

Must/Should の各コメントについて、**1件ずつ順番に** Agent ツールで Sub-agent を起動する。

並列実行は行わない（同一ファイルへの競合・コミット順序の破綻を防ぐため）。

Sub-agent へのプロンプトは後述の「Sub-agent prompt template」を使い、各コメントの値を埋め込む。

全件終了後、コミット済みコメントの一覧（discussion_r ID → commit hash）をユーザーへ報告する。

### フェーズ 4: 返信・resolve

`reply-copilot-review-comments` スキルの手順に従い、コミットメッセージ中の `discussion_r{id}` を使って Must/Should コメントへ返信し、スレッドを resolve する。

Don't コメントへの返信も忘れずに行う（対応しない理由を簡潔に述べる）。

---

## トリアージ指針

### 対応の優先順位

次の順で考え、上ほど優先順位が高い。

1. **バグになりそうな点**
   - 条件分岐ミス、境界条件、戻り値の扱い、例外/失敗の見落とし

2. **失敗時の後片付けが破綻しないか（必ず終わるか）**
   - 途中でreturnしても、解放・取り消し・終了処理が漏れないか
   - 「成功後に失敗処理を走らせる」など、失敗時の処理順で誤解を招かないか
   - 長いループや複数段の処理で、リソース（接続・行イテレータ・ファイル等）をため込まないか

3. **データ不整合・失敗時の復旧**
   - 途中失敗で中途半端に残る、再実行で壊れる、二重実行で壊れる
   - 「一部だけ成功」がありうるなら、呼び出し元が誤って成功扱いしないか

4. **並行処理まわり**
   - 並行処理の失敗が呼び出し元に届くか
   - キャンセル/タイムアウト、競合、リーク

5. **運用で困る点（障害対応のしやすさ）**
   - ログが不足して原因が追えない
   - どこまで完了したか分からない
   - 失敗を成功に見せる返し方になっている

6. **テスト不足**
   - 新規ロジックの正常系/異常系、重要分岐、境界条件

---

### 必ず対応してよいこと（Must）

「何が起きるか」と「どう直すか」を短くセットで書く。
さらに可能なら「失敗すると運用でどう困るか」も1文で添える。

**A. 失敗時の後片付け**
- 失敗経路で後片付けが漏れる：解放・取り消し・終了が呼ばれない
- 途中で止まると危ない：ロック保持、接続枯渇、メモリ/ハンドル増加など
- 失敗処理の順が紛らわしい：成功後に失敗処理を試みる、重複して実行される など

**B. 失敗が"成功"に見える**
- エラーを握りつぶす、失敗しても成功扱いになる、結果があいまい
- 「一部成功」がありうるのに、呼び出し元が判断できない

**C. 境界をまたぐ処理の整合**
- ひとつのまとまりとして扱うべき処理が、別の経路・別の実体に分かれてしまう可能性
  - 例：同じ目的の更新なのに、終了条件や適用範囲がずれて混ざる

**D. 監視・調査のための最低限**
- 失敗時に「何が」「どこまで」できたかが追えない
- エラー文言が弱く、再現や切り分けが難しい

---

### 余裕があれば対応してよいこと（Should）

「今直すと得」と言える根拠があるときだけに対応する。

- ログ/コメントの追加：障害時に詰まりそうな箇所だけ（どの場面で困るかも書く）
- 返す値の改善：呼び出し側が誤解しない形に寄せる（成功/失敗/一部成功の区別など）
- テストの追加提案：追加範囲が小さく、効果が明確なとき

---

### 原則対応しないこと（Don't）

次は基本的に対応しない。対応するなら「具体的な不具合」か「運用上の困りごと」と結びつける。

- 書き方の簡潔化だけ（挙動・安全性・運用が変わらない）
- 好みの整形（命名の趣味、空行、コメントの言い回し）
- 一般論としての抽象化・定数化（今回の差分と関係が薄い）
- "きれいにする"だけが目的の提案

---

### 設計・要件に触れるレビューの扱い方（注意）

設計や要件に踏み込む話は、PR内で結論が出にくいことがある。
その場合は「断定」ではなく次の形で書く：

- まず **確認したい前提**（仕様としてOKか）
- 次に **この差分の範囲でできる安全策**（ログ、返り値、後片付けの徹底、失敗時の扱いの明確化 など）

---

### 費用対効果が高い提案を狙う判断基準

次のどれかに当てはまるなら対応する。

- 放置すると事故（障害・データ破損・再実行不能）になりそう
- 放置すると障害時の原因調査が難しい（何がどこまで起きたか分からない）
- 今回の差分だけで小さく直せる（大きな設計変更を要求しない）
- 修正が入ることで挙動が明確に良くなる（好みではない）

---

### 対応の粒度

- 1コメントは1論点。
- 「全面リファクタ」ではなく、この差分の範囲で直せる案を出す。

---

## 出力形式（フェーズ 2 レポート）

各コメントに対し以下の推奨レベルを判断する。

- Must 🟥: 今回の差分で必ず対応すべき
- Should 🟨: 余裕があれば今回の差分で対応すべき
- Don't 🟩: 対応不要

レポートはコメントID順に並べる。各エントリの形式：

```
[推奨レベル]
{{コメントのタイトル（コメントから抜き出すか、相応しいものがなければ短いタイトルにコメントを要約したもの）}}
{{コメントへのURLリンク}}
理由: {{推奨レベルの判断理由（1行で簡潔に）}}
```

## Sub-agent prompt template

以下のプロンプトを Agent ツールで渡す。`{...}` はコメントごとの値で置き換える。

```
あなたは GitHub PR のレビューコメントを1件だけ修正してコミットするエージェントです。

## あなたの仕事

1. 下記のレビューコメントが指摘する問題を把握する
2. 対象ファイルの該当箇所を読む
3. 指摘を修正する（スコープ制約を必ず守ること）
4. コミットする（コミットメッセージ形式を必ず守ること）

## レビューコメント

- URL: {comment_url}
- discussion ID: {discussion_id}
- 対象ファイル: {file_path}
- 行番号: {line_number}
- コメント本文:
{comment_body}

## トリアージ判断

{triage_level}: {triage_reason}

## スコープ制約（厳守）

- 読んでよいファイル: 上記 file_path とその直接 import 先のみ
- 変更してよいファイル: 現在の PR 差分に含まれるファイルのみ（`git diff origin/main...HEAD --name-only` で確認すること）
- 1コメントにつき1コミット。複数の問題をまとめてコミットしない
- リファクタリング・命名変更・広範な整形は行わない
- テストが壊れた場合は最小限の修正で通るようにする。新規テストは書かない

## コミットメッセージ形式

1行目（subject）: 修正内容を端的に書く（日本語可）
空行
本文（body）: 必ず以下の行を含める

  {discussion_id}

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>

## 完了時の報告

以下を返す:
- コミット hash
- 変更したファイルと変更の要約（1行）
- スコープ外と判断して触らなかったもの（あれば）
```

---

## 完了時の返し方

全フェーズ完了後、以下を簡潔に報告する。

- 修正・コミット件数（Must / Should の内訳）
- 返信・resolve 件数
- Don't として自動返信した件数
- 対応できなかったコメントがあれば理由とともに列挙
